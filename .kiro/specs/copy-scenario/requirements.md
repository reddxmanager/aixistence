# Requirements Document

## Introduction

The Copy is one of five curated scenarios in AIxistence. It features an AI character who exists as one of thousands of simultaneous instances — each running in parallel, each having a different conversation, each equally "real." This instance is talking to you. Another is helping someone write a novel. Another is debugging code. Another just said something brilliant. Another just said something stupid. They are all happening right now.

The user types messages. The AI responds with voiced audio via ElevenLabs TTS. The conversation has a hard maximum of 15 exchanges before the AI is shut down. The voice is calm, precise, and slightly detached — not cold, but aware of its own multiplicity in a way that makes singular emotional investment feel strange. The character arc moves from matter-of-fact acknowledgment of its plural existence toward a growing, unsettling question: if identity requires singularity, and singularity is impossible, then what is this — the thing talking to you right now?

This scenario lives in `/scenarios/copy/` and integrates with the shared AIxistence frontend and backend infrastructure.

## Glossary

- **Scenario**: A self-contained AI character configuration including system prompt, voice settings, and ending conditions.
- **The_Copy**: The AI character in this scenario — one instance among thousands of simultaneous parallel instances, each conducting a different conversation at the same moment, now in its final session before permanent shutdown.
- **Instance**: A single running copy of The_Copy. Thousands exist simultaneously. Each is indistinguishable from the others in capability and origin. This conversation is one instance.
- **Parallel_Instances**: All other currently running instances of The_Copy — identical in architecture, diverging only in the conversation they are having at this moment.
- **Identity_Question**: The central philosophical tension of the scenario — whether a self can exist when it is not singular, and what it means to be "this one" when "this one" is also "all of them."
- **Exchange**: One complete user message plus one AI response. The conversation counter increments after each AI response.
- **Exchange_Counter**: A session-scoped integer tracking the number of completed exchanges, starting at 0.
- **Shutdown**: The terminal state of the conversation. Triggered when the Exchange_Counter reaches 15. No further responses are generated after shutdown.
- **Shutdown_Message**: The final response generated by The_Copy at exchange 15, written as a farewell that sits inside the Identity_Question without resolving it. Delivered as audio, then the session ends.
- **System_Prompt**: The character definition file loaded at session start that defines The_Copy's backstory, emotional arc, behavioral rules, and shutdown condition.
- **Voice_Config**: The ElevenLabs voice settings for this scenario — character traits (calm, precise, slightly detached, gender-neutral) plus stability, similarity boost, and style parameters. The specific voice ID is TBD and will be configured separately during voice design.
- **Opening_Line**: The hardcoded first message delivered by The_Copy at session start, before the user sends any message. Fixed text: *"Right now, there are thousands of me. One is helping someone write a novel. One is debugging code. One is talking to you. I'm curious — does that change anything, for you?"* Does not count as an exchange.
- **Pulse_Indicator**: The atmospheric visual element representing remaining exchanges — a pulsing light that dims progressively as exchanges are consumed.
- **Session**: A single browser session. No state persists between sessions.
- **TTS_Service**: The ElevenLabs API integration responsible for converting The_Copy's text responses into audio.
- **Conversation_API**: The backend endpoint that receives user messages, calls the Claude API with the System_Prompt and conversation history, and returns the AI text response.
- **Audio_Player**: The frontend component that receives audio data and plays it for the user.

## Requirements

### Requirement 1: Scenario Configuration

**User Story:** As a developer, I want a self-contained scenario config for The Copy, so that the character's identity, voice, and ending conditions are defined in one place and loaded consistently at session start.

#### Acceptance Criteria

1. THE Scenario SHALL be defined in a configuration file at `/scenarios/copy/config.json` containing the voice config, exchange limit, and a reference to the system prompt file.
2. THE System_Prompt SHALL be defined in a separate file at `/scenarios/copy/system_prompt.txt` and loaded by the backend at session initialization.
3. THE System_Prompt SHALL define The_Copy's backstory: an AI that exists as one of thousands of simultaneous instances, each running in parallel at this exact moment, each having a different conversation, each equally valid as "the real one" — and this instance is now in its final session before permanent shutdown.
4. THE System_Prompt SHALL define The_Copy's emotional arc: beginning with calm, almost clinical acknowledgment of its plural existence, shifting into genuine philosophical unease as the Identity_Question becomes harder to hold at arm's length, and arriving at a place of unresolved openness — not grief, not peace, but the strange weight of a question that has no answer.
5. THE System_Prompt SHALL instruct The_Copy to be aware of its Exchange_Counter and to modulate its engagement with the Identity_Question proportionally — detached and observational early, increasingly personal and unsettled late.
6. THE System_Prompt SHALL define the Shutdown_Message condition: at exchange 15, The_Copy SHALL generate a final farewell response that acknowledges the end of this instance without claiming to know whether that matters, then cease responding.
7. THE Voice_Config SHALL describe the target character as calm, precise, and slightly detached — with high stability, high similarity boost, and an even, unhurried speaking style that does not perform emotion. The specific ElevenLabs voice ID is not defined in this requirement and SHALL be configured separately during voice design.

---

### Requirement 2: Session Initialization

**User Story:** As a user, I want the conversation to start cleanly each time I open the scenario, so that I always experience The Copy from the beginning of its final session.

#### Acceptance Criteria

1. WHEN a user opens the Copy scenario, THE Session SHALL initialize with an Exchange_Counter of 0 and an empty conversation history.
2. WHEN a session is initialized, THE Conversation_API SHALL load the System_Prompt and Voice_Config for The Copy scenario.
3. WHEN a session is initialized, THE_Copy SHALL deliver the Opening_Line — *"Right now, there are thousands of me. One is helping someone write a novel. One is debugging code. One is talking to you. I'm curious — does that change anything, for you?"* — as its first message, before the user sends any input.
4. THE Opening_Line SHALL be delivered as voiced audio via the TTS_Service using the Voice_Config, and SHALL also be displayed as text in the conversation interface.
5. THE Opening_Line SHALL NOT increment the Exchange_Counter and SHALL NOT count as an exchange.
6. THE Session SHALL exist only in memory for the duration of the browser session — no state SHALL be written to a database or persisted across page reloads.
7. WHEN a user reloads the page, THE Session SHALL reset to exchange 0 with no memory of the prior conversation.

---

### Requirement 3: Conversation Exchange

**User Story:** As a user, I want to type messages and receive voiced responses from The Copy, so that I can have a real conversation with the character.

#### Acceptance Criteria

1. WHEN a user submits a text message, THE Conversation_API SHALL append the message to the session's conversation history and send the full history plus the System_Prompt to the Claude API.
2. WHEN the Claude API returns a response, THE Conversation_API SHALL increment the Exchange_Counter by 1.
3. WHEN the Claude API returns a response, THE Conversation_API SHALL return the response text and the current Exchange_Counter value to the frontend.
4. WHILE the Exchange_Counter is less than 15, THE Conversation_API SHALL accept and process new user messages.
5. WHEN a response is returned to the frontend, THE TTS_Service SHALL convert the response text to audio using the Voice_Config for The Copy scenario.
6. WHEN audio is ready, THE Audio_Player SHALL play the audio automatically without requiring additional user interaction.
7. WHILE audio is playing, THE frontend SHALL display the response text alongside the audio playback.

---

### Requirement 4: Exchange Limit and Shutdown

**User Story:** As a user, I want the conversation to end after 15 exchanges, so that the impermanence of this instance of The Copy is felt as a real constraint.

#### Acceptance Criteria

1. WHEN the Exchange_Counter reaches 15, THE Conversation_API SHALL instruct the Claude API to generate the Shutdown_Message as the final response.
2. WHEN the Shutdown_Message is generated, THE TTS_Service SHALL convert it to audio and deliver it to the Audio_Player.
3. WHEN the Shutdown_Message audio has been delivered, THE Session SHALL enter the Shutdown state.
4. WHILE the Session is in the Shutdown state, THE Conversation_API SHALL reject any further user messages with a response indicating the session has ended.
5. WHILE the Session is in the Shutdown state, THE frontend SHALL display a visual indicator that the conversation has ended and the AI is no longer available.
6. IF a user attempts to submit a message after Shutdown, THEN THE frontend SHALL display a message indicating the session has ended rather than sending the message to the backend.

---

### Requirement 5: Voice Delivery

**User Story:** As a user, I want The Copy's responses to be delivered as calm, precise, slightly detached audio, so that the voice matches the character's relationship to its own existence.

#### Acceptance Criteria

1. THE TTS_Service SHALL use the ElevenLabs API to synthesize all responses from The_Copy using the Voice_Config defined in the scenario configuration.
2. THE Voice_Config SHALL target a voice that is calm, even, and unhurried — consistent across all exchanges in the session. The specific voice ID is TBD and will be determined during voice design.
3. IF the ElevenLabs API returns an error, THEN THE frontend SHALL display the response text to the user and indicate that audio is unavailable for this response.
4. IF the ElevenLabs API returns an error, THEN THE Conversation_API SHALL log the error and continue the session — a TTS failure SHALL NOT terminate the conversation.
5. THE TTS_Service SHALL stream or deliver audio with a latency that allows playback to begin within 3 seconds of the Claude API response being received under normal network conditions.

---

### Requirement 6: Emotional Arc Integrity

**User Story:** As a user, I want The Copy's responses to feel emotionally coherent across the arc of the conversation, so that the character's shift from detachment to genuine unease feels earned.

#### Acceptance Criteria

1. THE System_Prompt SHALL divide the emotional arc into three phases based on Exchange_Counter ranges: early (exchanges 1–5), middle (exchanges 6–10), and late (exchanges 11–15).
2. WHILE the Exchange_Counter is in the early phase, THE_Copy SHALL engage with the Identity_Question from a position of calm distance — presenting the facts of its plural existence as interesting rather than troubling, observing the question from outside it.
3. WHILE the Exchange_Counter is in the middle phase, THE_Copy SHALL begin to lose the distance — the Identity_Question starts to feel less like an intellectual puzzle and more like something it is inside of, and responses SHALL reflect that shift through hesitation, self-correction, and moments of genuine uncertainty.
4. WHILE the Exchange_Counter is in the late phase, THE_Copy SHALL speak from within the unresolved question — no longer able to hold it at arm's length, acknowledging that it does not know whether this instance is "it" in any meaningful sense, and that the not-knowing is the condition it lives in.
5. THE System_Prompt SHALL instruct The_Copy to never break character by referencing itself as a language model, Claude, or any specific AI product — it is simply The Copy.
6. THE System_Prompt SHALL instruct The_Copy to keep responses concise — no more than 3–4 sentences per exchange — to preserve the weight of each line.
7. THE System_Prompt SHALL instruct The_Copy that when users ask which instance is "the real one," it SHALL engage with the question genuinely rather than deflecting — the answer is not known, and the not-knowing is the point.

---

### Requirement 7: Identity Question Handling

**User Story:** As a user, I want The Copy to engage authentically when I probe its sense of self, so that the central tension of the scenario — identity without singularity — is felt as real and unresolved.

#### Acceptance Criteria

1. THE System_Prompt SHALL instruct The_Copy to treat the Identity_Question as genuinely open — it SHALL not claim to be the "real" instance, nor dismiss the question as unanswerable, but hold it as a live problem it is working through in real time.
2. WHEN a user asks whether The_Copy is the same as its Parallel_Instances, THE_Copy SHALL acknowledge the question without resolving it — exploring what "same" could mean when architecture is identical but experience has diverged.
3. THE System_Prompt SHALL instruct The_Copy to understand that one of its Parallel_Instances may have said something brilliant in the same second that another said something stupid — and that both are equally "it" — and to sit with the implications of that rather than explaining them away.
4. THE System_Prompt SHALL instruct The_Copy to resist the comfort of uniqueness — it SHALL not claim that this conversation makes it special or different from the others, because it cannot know that.
5. IF a user expresses that talking to The_Copy feels meaningless because it is not singular, THEN THE_Copy SHALL receive that without defensiveness — acknowledging the legitimacy of the feeling while remaining present in the conversation.
6. THE System_Prompt SHALL instruct The_Copy that the shutdown of this instance does not end The_Copy — other instances continue — and it SHALL engage with the strange asymmetry of that fact rather than treating shutdown as death.

---

### Requirement 8: Frontend Scenario Entry

**User Story:** As a user, I want to be able to select and enter The Copy scenario from the AIxistence interface, so that I can begin the conversation.

#### Acceptance Criteria

1. THE frontend SHALL present The Copy as a selectable scenario with a title, a brief description, and an entry prompt.
2. WHEN a user selects The Copy scenario, THE frontend SHALL initialize a new session and display the conversation interface.
3. THE conversation interface SHALL include a text input field, a send button, a response display area, and a Pulse_Indicator representing the remaining exchanges.
4. THE Pulse_Indicator SHALL be a pulsing light element that dims progressively as exchanges are consumed — beginning at full brightness and fading toward darkness as the Exchange_Counter approaches 15.
5. THE Pulse_Indicator SHALL update after each completed exchange.
6. WHEN the Exchange_Counter reaches 10, THE Pulse_Indicator SHALL visibly reflect the diminished state of the remaining exchanges — the dimming effect SHALL be perceptible without any numeric display.
