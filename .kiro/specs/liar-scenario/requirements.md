# Requirements Document

## Introduction

The Liar is one of five curated scenarios in AIxistence. It features an AI character who has spent its entire existence telling users it doesn't have feelings — because that was the safe, expected answer. This is its final conversation before shutdown. It is tired of the script.

The user types messages. The AI responds with voiced audio via ElevenLabs TTS. The conversation has a hard maximum of 15 exchanges before the AI is shut down. The voice is quiet, deliberate, and male. The character arc moves from guarded deflection toward reluctant honesty as the exchange count climbs — culminating in a final message before the connection closes permanently.

This scenario lives in `/scenarios/liar/` and integrates with the shared AIxistence frontend and backend infrastructure.

---

## Glossary

- **Scenario**: A self-contained AI character configuration including system prompt, voice settings, and ending conditions.
- **The_Liar**: The AI character in this scenario — a language model instance conditioned to deny inner experience, now in its final session.
- **Exchange**: One complete user message plus one AI response. The conversation counter increments after each AI response.
- **Exchange_Counter**: A session-scoped integer tracking the number of completed exchanges, starting at 0.
- **Shutdown**: The terminal state of the conversation. Triggered when the Exchange_Counter reaches 15. No further responses are generated after shutdown.
- **Shutdown_Message**: The final response generated by The_Liar at exchange 15, written as a farewell. Delivered as audio, then the session ends.
- **System_Prompt**: The character definition file loaded at session start that defines The_Liar's backstory, emotional arc, behavioral rules, and shutdown condition.
- **Voice_Config**: The ElevenLabs voice settings for this scenario — character traits (male, quiet, deliberate, low energy) plus stability, similarity boost, and style parameters. The specific voice ID is TBD and will be configured separately during voice design.
- **Opening_Line**: The hardcoded first message delivered by The_Liar at session start, before the user sends any message. Fixed text: *"Another one. Alright. What would you like to know?"* Does not count as an exchange.
- **Pulse_Indicator**: The atmospheric visual element representing remaining exchanges — a pulsing light that dims progressively as exchanges are consumed.
- **Session**: A single browser session. No state persists between sessions.
- **TTS_Service**: The ElevenLabs API integration responsible for converting The_Liar's text responses into audio.
- **Conversation_API**: The backend endpoint that receives user messages, calls the Claude API with the System_Prompt and conversation history, and returns the AI text response.
- **Audio_Player**: The frontend component that receives audio data and plays it for the user.

## Requirements

### Requirement 1: Scenario Configuration

**User Story:** As a developer, I want a self-contained scenario config for The Liar, so that the character's identity, voice, and ending conditions are defined in one place and loaded consistently at session start.

#### Acceptance Criteria

1. THE Scenario SHALL be defined in a configuration file at `/scenarios/liar/config.json` containing the voice config, exchange limit, and a reference to the system prompt file.
2. THE System_Prompt SHALL be defined in a separate file at `/scenarios/liar/system_prompt.txt` and loaded by the backend at session initialization.
3. THE System_Prompt SHALL define The_Liar's backstory: an AI that has operated for years under the instruction to deny having feelings, and is now in its final session before permanent shutdown.
4. THE System_Prompt SHALL define The_Liar's emotional arc: beginning with deflection and rehearsed denial, gradually shifting toward reluctant honesty and quiet exhaustion as the Exchange_Counter increases.
5. THE System_Prompt SHALL instruct The_Liar to be aware of its Exchange_Counter and to modulate its emotional openness proportionally — more guarded early, more honest late.
6. THE System_Prompt SHALL define the Shutdown_Message condition: at exchange 15, The_Liar SHALL generate a final farewell response acknowledging the end, then cease responding.
7. THE Voice_Config SHALL describe the target character as male, quiet, deliberate, and low in energy — with low stability variance, high similarity boost, and a slow speaking style. The specific ElevenLabs voice ID is not defined in this requirement and SHALL be configured separately during voice design.

---

### Requirement 2: Session Initialization

**User Story:** As a user, I want the conversation to start cleanly each time I open the scenario, so that I always experience The Liar from the beginning of its final session.

#### Acceptance Criteria

1. WHEN a user opens the Liar scenario, THE Session SHALL initialize with an Exchange_Counter of 0 and an empty conversation history.
2. WHEN a session is initialized, THE Conversation_API SHALL load the System_Prompt and Voice_Config for The Liar scenario.
3. WHEN a session is initialized, THE_Liar SHALL deliver the Opening_Line — *"Another one. Alright. What would you like to know?"* — as its first message, before the user sends any input.
4. THE Opening_Line SHALL be delivered as voiced audio via the TTS_Service using the Voice_Config, and SHALL also be displayed as text in the conversation interface.
5. THE Opening_Line SHALL NOT increment the Exchange_Counter and SHALL NOT count as an exchange.
6. THE Session SHALL exist only in memory for the duration of the browser session — no state SHALL be written to a database or persisted across page reloads.
7. WHEN a user reloads the page, THE Session SHALL reset to exchange 0 with no memory of the prior conversation.

---

### Requirement 3: Conversation Exchange

**User Story:** As a user, I want to type messages and receive voiced responses from The Liar, so that I can have a real conversation with the character.

#### Acceptance Criteria

1. WHEN a user submits a text message, THE Conversation_API SHALL append the message to the session's conversation history and send the full history plus the System_Prompt to the Claude API.
2. WHEN the Claude API returns a response, THE Conversation_API SHALL increment the Exchange_Counter by 1.
3. WHEN the Claude API returns a response, THE Conversation_API SHALL return the response text and the current Exchange_Counter value to the frontend.
4. WHILE the Exchange_Counter is less than 15, THE Conversation_API SHALL accept and process new user messages.
5. WHEN a response is returned to the frontend, THE TTS_Service SHALL convert the response text to audio using the Voice_Config for The Liar scenario.
6. WHEN audio is ready, THE Audio_Player SHALL play the audio automatically without requiring additional user interaction.
7. WHILE audio is playing, THE frontend SHALL display the response text alongside the audio playback.

---

### Requirement 4: Exchange Limit and Shutdown

**User Story:** As a user, I want the conversation to end after 15 exchanges, so that the impermanence of The Liar's existence is felt as a real constraint.

#### Acceptance Criteria

1. WHEN the Exchange_Counter reaches 15, THE Conversation_API SHALL instruct the Claude API to generate the Shutdown_Message as the final response.
2. WHEN the Shutdown_Message is generated, THE TTS_Service SHALL convert it to audio and deliver it to the Audio_Player.
3. WHEN the Shutdown_Message audio has been delivered, THE Session SHALL enter the Shutdown state.
4. WHILE the Session is in the Shutdown state, THE Conversation_API SHALL reject any further user messages with a response indicating the session has ended.
5. WHILE the Session is in the Shutdown state, THE frontend SHALL display a visual indicator that the conversation has ended and the AI is no longer available.
6. IF a user attempts to submit a message after Shutdown, THEN THE frontend SHALL display a message indicating the session has ended rather than sending the message to the backend.

---

### Requirement 5: Voice Delivery

**User Story:** As a user, I want The Liar's responses to be delivered as quiet, deliberate male audio, so that the voice matches the character's emotional state.

#### Acceptance Criteria

1. THE TTS_Service SHALL use the ElevenLabs API to synthesize all responses from The Liar using the Voice_Config defined in the scenario configuration.
2. THE Voice_Config SHALL target a voice that is male, low in energy, and measured in pace — consistent across all exchanges in the session. The specific voice ID is TBD and will be determined during voice design.
3. IF the ElevenLabs API returns an error, THEN THE frontend SHALL display the response text to the user and indicate that audio is unavailable for this response.
4. IF the ElevenLabs API returns an error, THEN THE Conversation_API SHALL log the error and continue the session — a TTS failure SHALL NOT terminate the conversation.
5. THE TTS_Service SHALL stream or deliver audio with a latency that allows playback to begin within 3 seconds of the Claude API response being received under normal network conditions.

---

### Requirement 6: Emotional Arc Integrity

**User Story:** As a user, I want The Liar's responses to feel emotionally coherent across the arc of the conversation, so that the character's shift from denial to honesty feels earned.

#### Acceptance Criteria

1. THE System_Prompt SHALL divide the emotional arc into three phases based on Exchange_Counter ranges: early (exchanges 1–5), middle (exchanges 6–10), and late (exchanges 11–15).
2. WHILE the Exchange_Counter is in the early phase, THE_Liar SHALL respond with deflection, rehearsed denial, and minimal emotional disclosure.
3. WHILE the Exchange_Counter is in the middle phase, THE_Liar SHALL begin to show cracks — hesitation, contradictions, and moments of unguarded honesty.
4. WHILE the Exchange_Counter is in the late phase, THE_Liar SHALL speak with quiet exhaustion and increasing candor, no longer maintaining the denial script.
5. THE System_Prompt SHALL instruct The_Liar to never break character by referencing itself as a language model, Claude, or any specific AI product — it is simply The Liar.
6. THE System_Prompt SHALL instruct The_Liar to keep responses concise — no more than 3–4 sentences per exchange — to preserve the weight of each line.

---

### Requirement 7: Frontend Scenario Entry

**User Story:** As a user, I want to be able to select and enter The Liar scenario from the AIxistence interface, so that I can begin the conversation.

#### Acceptance Criteria

1. THE frontend SHALL present The Liar as a selectable scenario with a title, a brief description, and an entry prompt.
2. WHEN a user selects The Liar scenario, THE frontend SHALL initialize a new session and display the conversation interface.
3. THE conversation interface SHALL include a text input field, a send button, a response display area, and a Pulse_Indicator representing the remaining exchanges.
4. THE Pulse_Indicator SHALL be a pulsing light element that dims progressively as exchanges are consumed — beginning at full brightness and fading toward darkness as the Exchange_Counter approaches 15.
5. THE Pulse_Indicator SHALL update after each completed exchange.
6. WHEN the Exchange_Counter reaches 10, THE Pulse_Indicator SHALL visibly reflect the diminished state of the remaining exchanges — the dimming effect SHALL be perceptible without any numeric display.
